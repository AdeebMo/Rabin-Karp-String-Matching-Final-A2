    Overview

    What problem are we solving?

    The string matching problem asks us to find every starting position in a text T where a pattern P appears exactly. If the text length is n and the pattern length is m, there are n - m + 1 possible windows to inspect.

    Naive Search

    The baseline method compares characters directly.

    Naive matching tries every alignment and checks characters one by one. It is simple and correct, but repeated mismatches can make it expensive because the same text positions may be compared many times.

    Rabin-Karp Main Idea

    Compare hashes before comparing characters.

    Rabin-Karp converts the pattern and each text window into numeric fingerprints. Most windows can be rejected quickly when their hash does not match the pattern hash.

    Polynomial Hashing

    Characters become weighted by powers of the base.

    The app shows both the polynomial form and the iterative modular form: hash(S) = (S[0] × d^(m-1) + ... + S[m-1]) mod q.

    Rolling Hash

    Slide the window without recomputing the whole hash.

    Rabin-Karp removes the outgoing character, shifts the remaining contribution by multiplying with d, then adds the incoming character. This makes each hash update O(1).

    Spurious Hits

    Hash collisions create false alarms.

    Two different strings can produce the same hash modulo q. That is why a hash match triggers character-by-character verification instead of immediate acceptance.

    Complexity

    Expected efficiency with a careful warning.

    Preprocessing the pattern and first window takes O(m). Sliding through all windows takes O(n - m + 1) hash checks. The expected running time is O(n + m), but worst-case behavior can still reach O(nm) if many spurious hits force repeated verification.

    Comparison with KMP

    Two linear-time ideas, two different mechanisms.

    KMP uses a prefix function to avoid rechecking characters after mismatches, giving deterministic O(n + m) time. Rabin-Karp is especially attractive when hashing can be reused for many patterns or many windows.

    Presentation Checklist

    Suggested talking points for the recorded presentation

    • First I enter the text and pattern.
    • Then the tool computes the pattern hash.
    • Next it computes the hash of the first text window.
    • The window slides one character at a time.
    • Instead of recomputing the hash, Rabin-Karp updates it using the rolling hash formula.
    • When the hash values match, the algorithm verifies the actual characters.
    • If the characters match, it is a real match.
    • If the hashes match but the characters do not, it is a spurious hit.
    • Finally, I compare the Rabin-Karp output with naive search to verify correctness.
    • The key structural property is the rolling hash invariant.
